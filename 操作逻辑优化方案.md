> 范围：歌曲长按快捷操作 · 队列拖拽排序+滑动删除 · 返回键/导航逻辑 · 全屏播放器手势/控制
> 状态：**方案稿（未写码）**，供确认后再实施
> 约束提醒：本工程是「Compose 孤岛」架构（每个 *Controller 自建 ComposeView），交互回调多数经 Java 外壳（MainActivity → NowPlayingEvent / LibraryEvent → *ActionsController）转发。下文每项都标注了**是否需要动 Java 层**，便于你决定先做哪些「纯 Compose、低风险」项。

## 一、歌曲长按快捷操作

### 现状
`TrackListScreen.kt` 的 `TrackRow`（:313）：
- **单击整行 = 立即播放**（`actions.onPlay.run()`，:321）。
- 行尾常驻一条「动作轨」（:369-389）：收藏心 `MiniIconBtn` + 条件性「加入歌单」按钮（`showPlaylistAction`）+ 条件性「更多」菜单（仅流媒体轨有 `onEdit/onDelete`，:386）。
- `TrackMoreMenu`（:406）只含「编辑/删除」两项，且**仅流媒体轨可见**。
- **没有任何长按（`combinedClickable` / `detectTapGestures(onLongPress)`）逻辑**。
- `actionRailWidth`（:394）按可见按钮数预留 35dp×N 宽度——按钮越多越挤占标题空间。

### 问题
1. **本地歌曲无法「下一首播放」或「加入队列」**——单曲级别的队列操作完全缺失，只能整列表播放，或走「加入歌单」对话框（3-4 步）。
2. **按钮墙占位**：每行常驻 1-3 个图标按钮，是桌面/鼠标 hover 思路；移动端标准是长按出上下文菜单。
3. **可发现性割裂**：编辑/删除藏在「更多」里，收藏/加歌单是独立按钮，操作入口不统一。

### 优化方案
- 行交互改为 `combinedClickable`：**单击=播放（不变）**，**长按=弹出底部操作面板（BottomSheet）**。
- 底部面板统一收纳单曲操作：`下一首播放` · `加入队列` · `加入歌单` · `收藏/取消收藏` · （流媒体）`编辑` · `删除`。面板顶部显示该曲封面+标题，明确操作对象。
- 行尾常驻按钮**收敛为仅保留 1 个**：当前曲目用收藏心，或干脆移除常驻按钮、全部进长按面板（更干净，留白更多，契合「清新风格」目标）。建议折中：**只留收藏心常驻**（高频、需即时反馈），其余进长按面板。

### 涉及文件与改动点
- `ui/TrackListScreen.kt`：`TrackRow` 改 `combinedClickable`；新增一个可复用的 `TrackActionSheet`（Compose `ModalBottomSheet`）；`TrackRowActions` 扩展 `onPlayNext`、`onAddToQueue` 两个回调。
- 建议新建 `ui/TrackActionSheet.kt`，给列表/队列/全屏播放器共用，避免再造「菜单汤」。
- **Java 层（需要）**：`下一首播放` / `加入队列` 需新增 `LibraryEvent`（或现有事件扩展）→ `PlaybackActionsController` 调 `EchoPlaybackService` 的插队/入队 API。需确认 Service 是否已有 `addToQueue(index)` / `playNext`；若无需新增。`加歌单`、`收藏`、`编辑`、`删除`已有回调，直接复用。

### 风险
中。纯 UI（长按+面板）低风险；「下一首/加入队列」要打通到 Service，需核对 `EchoPlaybackService` 现有入队接口。建议先实现「长按面板复用现有回调」（零 Java 改动即可上线收藏/加歌单/编辑/删除），再单独补「下一首/加入队列」。

## 二、队列拖拽排序 + 滑动删除

### 现状
`ui/QueueScreen.kt`：
- 队列为只读 `LazyColumn`（:156），`itemsIndexed` key 用 `track.key`（:158，稳定 key，利于动画）。
- 每行底部一条按钮行（:226-241）：收藏、加入歌单、移除——**移除是按钮**，非侧滑。
- **无拖拽重排**。`PlaylistActionsController.moveSelectedPlaylistTrack` 仅服务于「歌单」编辑，队列没有对应 UI 与 Service 调用。

### 问题
1. 队列**不能调整播放顺序**，只能删了重加——这是播放器核心操作的明显缺失。
2. 删除靠点小按钮，移动端更顺手的是**侧滑删除**。

### 优化方案
- **拖拽排序**：用 Compose 官方 `LazyColumn` + `Modifier.draggable` 方案，或引入 `sh.calvin.reorderable`（成熟、轻量）。行首加「拖拽手柄」图标，长按手柄拖动重排。
- **侧滑删除**：行外包 `SwipeToDismissBox`（Material3），左滑露出删除背景，松手移除；保留底部「移除」按钮作为无障碍兜底（或移入长按面板，与第一节统一）。
- 拖拽过程用 key 动画（已具备稳定 key）使移动顺滑。

### 涉及文件与改动点
- `ui/QueueScreen.kt`：`QueueScreen` 引入 reorderable 状态；`QueueTrackRow` 加拖拽手柄 + `SwipeToDismissBox`。`QueueTrackActions` 扩展 `onMove(from,to)`。
- **Java 层（需要）**：拖拽落点提交需新增事件 → `PlaybackActionsController.moveQueueTrack(from,to)` → `EchoPlaybackService` 需提供 `moveQueueItem(from,to)`（Media3 `ExoPlayer.moveMediaItem` 可直接支持）。需核对 Service 当前是否暴露该能力。
- 依赖：如用第三方 reorderable 库，需加 Gradle 依赖（与「尽量纯 Compose」目标权衡；也可手写 `detectDragGesturesAfterLongPress` + 偏移，无新依赖但代码量更大）。

### 风险
中高。拖拽与 Media3 队列索引同步要小心：拖拽期间 UI 顺序与 Service 真值需在松手时一次性提交，避免每帧打 Service。当前曲目高亮（`current`）在重排后要保持正确。建议拖拽**仅本地预览**，`onDragEnd` 才调 `moveMediaItem`。

## 三、返回键 / 导航逻辑

### 现状
- **不是真实返回栈**，而是 `MainBackNavigationPolicy.resolve`（`MainBackNavigationPolicy.kt:4`）依据**扁平 route 状态**（selectedTab / networkPage / settingsPage / libraryGroupKey / playlistId）按**优先级顺序**计算「下一步退到哪」：库分组→歌单详情→设置子页→网络各子页→非首页 Tab→首页→不处理（退出 App）。
- 切换 Tab 时（`MainRouteController.navigateToTab`，:129）会把网络/设置子页**重置回各自首页**、库 HOME→SONGS。
- 全屏播放器的返回由 Java 外壳单独拦截（`NowPlayingOverlayController.isShowing()` 路由），不在该 policy 内。

### 问题
1. **历史丢失**：跨 Tab 的浏览路径无法逆序回退。例：首页→点某专辑→切到设置→返回，不会回到「专辑详情」，而是按 policy 跳设置首页/首页。用户的真实浏览轨迹不被记忆。
2. **Tab 切换即重置子页**：从网络三级页切去别的 Tab 再切回，落到网络首页，丢失上下文。
3. policy 的优先级是「就近父级」启发式，**深层嵌套**（如 `NETWORK_WEBDAV_SOURCE_TRACKS`）虽逐级配置，但任何新增页面都要手工往 policy 里加分支，易遗漏。

### 优化方案（两档，按投入选）
- **方案 A（小改，P1 可选）**：保留现有 policy，但把「Tab 切换重置子页」改为**记忆每个 Tab 的最后子页**（per-tab last route），切回时恢复。低风险、体验提升明显。
- **方案 B（重构，P2）**：引入**真实导航回退栈**（一个 `ArrayDeque<RouteState>`），每次导航 push、返回 pop，policy 退化为「栈空时是否退出」。可彻底解决逆序回退与新页面易遗漏问题。但牵动 Java 外壳 + `MainActivityViewModel` 路由持久化 + 现有 `MainBackNavigationPolicy` 的单元测试。

### 涉及文件与改动点
- 方案 A：`MainRouteController.kt`（`navigateToTab` 不再重置，改为查/存 per-tab 子页 map）+ `MainActivityRouteState` 增字段。
- 方案 B：新增 `RouteBackStack`（Kotlin），改 `MainRouteController` 全量导航入口走入栈/出栈，`applyBackNavigation` 改为 pop；更新/重写 `MainBackNavigationPolicy` 及其测试。
- **Java 层**：两方案都涉及 Java 外壳对路由状态的读写；B 改动面更大。

### 风险
A 低、B 高。B 需回归现有 32 个 instrumentation 测试与 `verify-release.ps1` 门禁中与导航相关的断言。**建议先上 A**，B 单独立项评估。

## 四、全屏播放器手势 / 控制

### 现状
`ui/NowPlayingOverlayController.kt`：
- 关闭**仅靠**左上 `ArrowDown` 圆钮（:321）或系统返回键；**无下滑关闭手势**。
- 进度条 `ScrubProgress`（:1073）：按下+拖拽 Seek（:1089），可点可拖；但**无 thumb 放大 / 时间气泡**等 scrub 反馈。
- **封面无横滑切歌**；上一首/下一首只能点按钮（`TransportControls`，:887）。
- 点封面 / 点歌词区 = 切换「封面↔歌词」视图（:360、:647）——点击切视图缺乏明显可发现性，且**易误触**（想看封面细节却翻到歌词）。
- **控制冗余**：右上「更多」菜单（收藏/加歌单/队列/切歌词，:962）与底部 `ModeControls`（收藏/随机/循环/队列，:934）**重叠**——收藏、队列两处都有。

### 问题
1. 缺「下滑关闭」是与主流播放器最大的直觉差距。
2. 切歌只能点按钮，封面横滑是普遍肌肉记忆。
3. 「点封面=翻歌词」易误触、无提示。
4. 收藏/队列入口重复，认知负担。

### 优化方案
- **下滑关闭**：根容器加竖向拖拽手势，下拉超过阈值/速度即 `onDismiss`，并随手指位移做内容下移+渐隐（与现有 `slideOutVertically` 退场衔接）。
- **封面横滑切歌**：封面区加横向拖拽，左滑=下一首、右滑=上一首（带回弹动画）；保留按钮。**注意与「下滑关闭」的手势方向冲突**——靠主轴方向判定（竖直优先关闭、水平优先切歌）。
- **翻歌词改为显式**：点封面切歌词改为**右上「歌词」图标按钮**或封面下方一个小「歌词」入口，封面点击不再翻页（或保留但加 hint）。
- **去冗余**：底部 `ModeControls` 专注「播放模式」（随机/循环 + 队列），把「收藏/加歌单」收进「更多」菜单或移到标题旁单一收藏心；两处别再重复收藏。
- **Seek 反馈**：scrub 时 thumb 放大 + 顶部显示目标时间气泡。

### 涉及文件与改动点
- 仅 `ui/NowPlayingOverlayController.kt`（**纯 Compose、无 Java 改动、构造签名不变**）：根 `Box` 加 `pointerInput` 竖滑关闭；`ArtworkSection` 加横滑回调（复用已有 `onPrevious/onNext`）；`ScrubProgress` 加 scrub 视觉；精简 `ModeControls`/`NowPlayingMoreMenu` 的项。
- 与上次「清新化」一致，**本文件可独立迭代、风险低**。

### 风险
低。主要难点是**多手势共存的方向判定**（下滑关闭 vs 横滑切歌 vs 进度条拖拽分区），需在不同区域限定手势作用域（进度条独占其 Y 带、封面独占横向、外层空白区竖向关闭）。

## 跨领域建议：统一的「曲目操作面板」组件

第一节（列表长按）、第二节（队列）、第四节（播放器更多菜单）都在各自重复定义曲目操作项，正是记忆里提到的「Controller/菜单汤」。建议**新建一个共享 `TrackActionSheet`（Compose ModalBottomSheet）**，集中「下一首/加入队列/加歌单/收藏/编辑/删除」，三处共用。这样既减步骤、统一视觉，也为后续清新化收敛了一处分散逻辑。
